Testing Speed

1) Right click inspect element to go into Chrome's DevTools
2) Go in the Network tab
3) Hold SHIFT so that everything gets redownloaded even if you have them already on your machine
4) Refresh the page
5) Investigate!

You can also do a new incognito window which will also make sure no cookies are used.

The pale part of the graph is the start when the browser started requesting the resource
The darker part is when the resource started being downloaded

The colour code is the filetype.

You can click on each filename to investigate if it has the proper cache settings.

Consider adding async attribute to your javascript calls if you only need javascript for user interaction AFTER the page is done loading.

Make sure your CSS comes before your Javascript

Timeline

The timeline tab allows you to figure out your frame per seconds. Click on the "recording" button to start recording, and then interact with your page. Stop the recording and go to the Frames tab. The page will be filled with colours:

Green: Painting
Purple: Rendering (style recalculation)
Yellow: Scripting
Blue: HTML

The highest the frame rate, the better.

CPU Profiling

Click on the Profile page and select "Collect Javascript CPU Profile" to investigate which script is slow. Click start and interact with your page, then click stop.

Finding Memory Leaks

Click on the Timeline tab and click on Memory.
Click recording, interact with the web page and stop the recording.
If you notice something that looks like memory leak, you would need to use Heap Snapshot

Go to the Profile tab and select Take Heap Snapshot
Interact with the site and take another snapshot (recording button)
Compare the two snapshots (arrow next to summary)
One sign of memory leak is under Detached DOM tree
